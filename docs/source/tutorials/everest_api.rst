********************
Using the EVerestAPI
********************

.. note::

  Find in depth explanations about the EVerestAPI :doc:`here <../explanation/adapt-everest/apis>`.

Introduction
============

The EVerestAPI is an API which is kept as stable as possible over different releases of EVerest.
It provides json-encoded data access via MQTT.
The application is providing a stable extension point for applications which are developed outside
the EVerest framework, but need to access or provide data from or to it.

Architecture
============

The EVerestAPI is specified as AsyncAPI 3.0 documents: :doc:`EVerest API Reference </reference/api/autogenerated_api_index>` .
The individual APIs more or less match the internal EVerest interfaces.
The APIs are implemented as standard EVerest modules.
As a rule-of-thumb, one internal interface is typically provided by one module.
Configuring the required APIs is therefore simply done by adding the corresponding modules to the EVerest configuration.
They are connected to other modules in the same way as usual.

Configuration
=============

The ``config/bringup/config-bringup-EVerestAPI-entrypoint.yaml`` configuration file demonstrates the API.
It does not implement any internal functionality, but provides a few API modules and BringUp modules to interact manually.

.. hint::

    It's advisable to set the access specification in the configuration to ``allow_global_read: true`` for each EVerestAPI module.
    This allows the modules to figure out if there are multiple EVerestAPI modules active and avoids that the initial ``ready_beacon``
    is sent multiple times.

Example
=======

Start some MQTT monitoring, like MQTT-Explorer first.

Start the example configuration:

.. code-block:: bash

    workspace $ ./run-scripts/run-bringup-EVerestAPI-entrypoint.sh

You will see the EVerest log and another two panes showing power supply control UIs.

Monitor the ``everest_api/#`` MQTT topic.
First a ``everest_api/ready_beacon`` with empty json load indicates, that some EVerestAPI is available.
After that, messages will arrive on other topics like ``everest_api/1/power_supply_DC/ps_dc_1/e2m/heartbeat``.

.. note::

    To exit the tmux session, press ``Ctrl+B`` and then ``d``.

Communication Monitoring
========================

At both ends, applications need to know, if the other side is still active.
The specific topic in the example above can be used by an API client to detect, if the API module ``ps_dc_1``, providing
a ``power_supply_DC`` API is still operational.
The heartbeat is a periodically increased integer as can be observed in the MQTT traffic easily.
The heartbeat period is defined in the manifest (default) or configuration file by ``cfg_heartbeat_interval_ms``.

To allow the EVerest API module to detect if the client is still alive, on can configure the ``cfg_communication_check_to_s`` to a value greater than 0.
It is the client applications responsibility to send ``true`` (raw, not enclosed in ``{}``) within this time to the corresponding topic periodically:

``everest_api/1/power_supply_DC/ps_dc_1/m2e/communication_check``

Otherwise, the API modules will raise errors within EVerest (``Error raised, type: generic/CommunicationFault, sub_type: ....``) but keep working on their part.

entrypoint_API
==============

The `entrypoint_API` helps to discover which API endpoints are available if the client does not know from other sources.

The afore-mentioned ``everest_api/ready_beacon`` is part of the `entrypoint_API` as well.

The most basic query to the `entrypoint_API` is sending the json below to ``everest_api/discover``:

.. code:: json

    {
        "headers": {
            "replyTo": "reply/to/address"
        }
    }

For the given example configuration this results in four individual answers, that the client needs to gather:

.. code:: json

    {
        "apis": [
            {
                "communication_monitoring": {
                    "communication_check_period_s": 10,
                    "heartbeat_period_ms": 10000
                },
                "module_id": "err_hist_1",
                "type": "error_history_consumer",
                "version": 1
            }
        ]
    }

    {
        "apis": [
            {
                "communication_monitoring": {
                    "communication_check_period_s": 10,
                    "heartbeat_period_ms": 10000
                },
                "module_id": "ps_dc_1",
                "type": "power_supply_DC",
                "version": 1
            }
        ]
    }

    {
        "apis": [
            {
                "communication_monitoring": {
                    "communication_check_period_s": 5,
                    "heartbeat_period_ms": 1000,
                    "request_reply_timeout_s": 550
                },
                "module_id": "dm_1",
                "type": "display_message",
                "version": 1
            }
        ]
    }

    {
        "apis": [
            {
                "communication_monitoring": {
                    "communication_check_period_s": 10,
                    "heartbeat_period_ms": 10000
                },
                "module_id": "ps_dc_2",
                "type": "power_supply_DC",
                "version": 1
            }
        ]
    }

The modules reply quickly from memory to the discover request.
Therefore a timeout of 1 second can be considered safe in order to make sure to get all module's replies.
Not the ``"apis"`` key holds an array.
While not the case at the time of writing, an API module could implement multiple APIs and therefore report them this way.

The alternative query to the `entrypoint_API` is sending the json below to a topic restricting the responders to power_supply_DC APIs: ``everest_api/query-modules/power_supply_DC``:

.. code:: json

    {
        "headers": {
            "replyTo": "reply/to/address"
        }
    }

The results are similar to the ones given above, but filtered by the selected API type.
